## About this organization

**Bedatty Engineering** is a **personal** space: a homelab-style collection of things I build, learn, and share—not a company or a product org with a roadmap and releases on a schedule.

**What you’ll find here**

- side projects and experiments;
- self-hosted tools, scripts, and automations;
- work-in-progress and “I’m tinkering with this” repos.

**How I work**

- **Build first, polish later**—repos may be rough, sparse on docs, or idle for a while.
- **No deadlines, no stakeholders**—pace is curiosity-driven; activity comes in bursts.
- **Each repository is its own world**—stack, maturity, and whether PRs are practical can differ; check that repo’s readme and issue tracker for specifics.

**What that means for you**

- There is **no SLA** on issues or reviews (see also `SUPPORT.md`).
- Software is provided **as-is**; don’t rely on these projects for critical production without your own evaluation.
- **Feedback and improvements are still welcome**—ideas in issues, and pull requests where the repo supports them (see below).

---

## If you are not the maintainer

- **Ideas and suggestions**: please open an **issue** if you have ideas to improve a project, report a problem, or ask a focused question. I appreciate thoughtful input even when I cannot act on it immediately.
- **Pull requests**: if a repository is set up for contributions (e.g. clear setup, tests, or a note in its own readme), **PRs are welcome** for small, focused changes. For larger work, opening an issue first helps avoid wasted effort. I may not merge every PR, but I will consider them when they fit the project.
- You are always welcome to **fork** a repository and adapt it under that repository’s license.
- For security-sensitive matters, use the process described in `SECURITY.md` when applicable.
- Be respectful and follow the organization’s [`CODE_OF_CONDUCT.md`](CODE_OF_CONDUCT.md).

---

## If you are the maintainer

Use whatever workflow fits each repo (branches, PRs, tests). Keep changes small and documented when the context is not obvious.

### License defaults *(heuristic, not legal advice)*

Use this when picking or standardizing a license for a **new** repo. **Category → license:**

| Repository category | Typical examples | Default license | Why (short) |
|---------------------|------------------|-----------------|-------------|
| **Utilities, Actions, libraries** | CLI helpers, shared npm/go libs, **composite actions**, reusable workflow *definitions* shipped as repo content | **MIT** | Permissive; low friction for Actions and small tools. |
| **Shareable infra / IaC** | **Terraform modules**, **CDK** constructs/stacks meant for reuse, published infra kits | **Apache 2.0** | Common in infra; explicit **patent grant**. |
| **Anti–closed-commercial** *(optional)* | Code you don’t want taken into proprietary products **without** reciprocity | **GPL** *(rare)* | Copyleft—usually **not** worth the overhead for hobby repos. |

Each project still needs its own root **`LICENSE`**; forks and PRs follow **that** file, not this table.
